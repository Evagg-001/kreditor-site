# KREDITOR Platform Architecture

## 1. Назначение

KREDITOR Platform — цифровая экосистема для сопровождения клиентов по вопросам кредитных обязательств, судебных споров, взыскания задолженности и защиты прав должников и кредиторов.

Платформа развивается без остановки действующего публичного сайта.

## 2. Архитектурная стратегия

Используется поэтапная архитектура:

- публичный сайт сохраняется на HTML, CSS и JavaScript;
- серверная часть создаётся как отдельная система;
- взаимодействие происходит только через защищённый API;
- CRM, личный кабинет и документы не размещаются в публичном контуре;
- критические данные не хранятся в браузере пользователя.

## 3. Основные контуры

### 3.1 Public Website

Публичная часть:

- страницы услуг;
- статьи и база знаний;
- формы первичного обращения;
- SEO-контент;
- юридические документы;
- контактная информация.

Публичный сайт не имеет прямого доступа к базе данных CRM.

### 3.2 API Gateway

Единая точка входа для запросов сайта и будущих приложений.

Функции:

- проверка входных данных;
- ограничение количества запросов;
- защита от автоматических атак;
- проверка происхождения запросов;
- журналирование;
- маршрутизация запросов во внутренние сервисы.

### 3.3 Lead Service

Сервис обработки обращений.

Функции:

- приём заявок;
- серверная валидация;
- фиксация согласия пользователя;
- присвоение идентификатора обращения;
- безопасная передача заявки в CRM;
- защита от повторной отправки и спама.

### 3.4 Identity and Access Service

Сервис авторизации и управления доступом.

Функции:

- учётные записи;
- многофакторная аутентификация;
- управление сессиями;
- роли и права доступа;
- блокировка подозрительной активности;
- журнал входов.

### 3.5 CRM

Закрытая внутренняя система.

Функции:

- карточки клиентов;
- обращения;
- задачи;
- статусы;
- сотрудники;
- коммуникации;
- история действий;
- контроль сроков.

### 3.6 Client Portal

Личный кабинет клиента.

Функции:

- просмотр статуса обращения;
- защищённая переписка;
- загрузка документов;
- уведомления;
- история взаимодействия;
- управление согласиями.

### 3.7 Document Vault

Защищённое хранилище документов.

Требования:

- шифрование;
- разграничение доступа;
- журнал скачиваний и изменений;
- антивирусная проверка;
- ограниченные ссылки;
- резервное копирование.

### 3.8 AI Service

Изолированный интеллектуальный сервис.

Функции:

- первичная классификация обращений;
- поиск по базе знаний;
- подготовка рекомендаций;
- формирование черновиков документов;
- передача сложных случаев специалисту.

AI не принимает окончательные юридические решения без контроля специалиста.

### 3.9 Audit Service

Неизменяемый журнал событий.

Фиксирует:

- входы пользователей;
- изменение ролей;
- просмотр данных;
- загрузку документов;
- изменение статусов;
- административные действия;
- события безопасности.
